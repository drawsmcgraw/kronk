# Incident Post-Mortem — 2026-05-31 Kronk Host Hang

**Status:** Resolved. Mitigations in place.
**Severity:** High — full host hang, hard reboot required.
**Date:** 2026-05-31
**Detected at:** ~15:01 EDT (system stopped responding; operator noticed
through a Claude harness crash in their IDE)
**Recovery:** Hard reboot at 15:07 EDT. All services came back via
existing `restart: unless-stopped` policies + Docker daemon enabled at
boot. ~6 min downtime.

---

## TL;DR

Kronk hung hard while Music Assistant was streaming audio to a Sonos Move.
Root cause was **system memory pressure** on the host, with the iGPU's GTT
(Graphics Translation Table) memory pool taking up 70+ GB of the 122 GB
system RAM via permanently-loaded llama.cpp models — many of which were
unused. Music streaming was the straw that pushed the immediately-available
memory pool below the threshold where the kernel could keep up. No clean
OOM kill, no panic — just a hard hang. Bun (the Claude harness runtime)
was the loudest casualty because of its W+X JIT pages, but the entire host
was sick.

**Mitigation:** Stopped + disabled three unused llama.cpp models
(`talkie`, `devstral-q4`, `mistral-nemo`), freeing **55 GB of GTT**.
Added free-RAM + GTT visibility to the `/resources` dashboard with
color-coded thresholds. Memory watchdog notification deferred pending
operator choice of notification mechanism.

## Symptoms

### Bun (Claude harness) crash visible to operator

```
panic(main thread): Segmentation fault at address 0x7F6817209758
oh no: Bun has crashed.
…
oh no: multiple threads are crashing
panic: Segmentation fault at address 0x0
Illegal instruction (core dumped)
```

Multi-thread segfaults at first one specific address, then `0x0`. This
shape is characteristic of **JIT pages getting scribbled on** by something
upstream — not a normal application bug. Bun's JSC engine maps W+X pages
for compiled JavaScript; under memory pressure or amdgpu allocator misbehaviour,
those pages can be corrupted while still in use.

### Journal stops abruptly

`journalctl -b -1` shows routine health-probe loops continuing through
15:01:41 and then **nothing** — no shutdown sequence, no panic message,
no OOM kill record. That pattern is the kernel hung so hard that
systemd-journald couldn't flush the buffer. Consistent with a deep lockup
in memory management or amdgpu driver.

### No kernel panic or MCE

Searched kernel log for the last hour before the crash. No panics, no
machine-check exceptions, no thermal trips, no hardware errors. Pure
software-stack failure.

## Root cause analysis

### Primary: System memory tightened progressively under combined load

From `sar -r` snapshots taken every 10 min:

| Time | `kbmemfree` | `kbavail` | `%memused` |
|---|---|---|---|
| 14:10 | 4.7 GB | 35 GB | 70% |
| 14:30 | 2.5 GB | 34 GB | 70% |
| 14:50 | 2.4 GB | 34 GB | 70% |
| **15:00** | **1.7 GB** | **34 GB** | **70%** |
| 15:01:41 | (hard hang — no further data) | | |

The `%memused` and `kbavail` (memory available after reclaim, including
page cache) stayed apparently healthy at 70% / 34 GB — so naive monitoring
would have shown the system as "fine." But `kbmemfree` — the
**immediately-available pool** without page-cache reclaim — was dropping
steadily, and the **page-fault rate doubled** in the 10 minutes before the
hang (2,729 → 5,205 faults/sec). Both signals the kernel was working
harder and harder to keep up with allocation requests.

### Contributing factor #1: Permanently-loaded llama.cpp models we weren't using

The iGPU on this machine (Radeon 8060S) has no dedicated VRAM — it
allocates from system RAM via GTT, configured for up to ~101 GB. At the
time of the hang, the following were permanently loaded:

| Service | GTT footprint | Used by |
|---|---|---|
| `llama-gemma3-4b` | ~3 GB | router (active) |
| `llama-gemma4-e4b` | ~5 GB | coordinator + agents (active) |
| `llama-devstral-q4` | ~14 GB | coding/devops agents (**unused — operator uses separate harness**) |
| `llama-mistral-nemo` | ~13 GB | unassigned (loaded for the v2-coord bench 2026-05-30, never unloaded) |
| `llama-talkie` | ~13 GB | talkie agent (**rarely invoked**) |
| `llama-bonsai` | ~1 GB | router fallback |
| `wyoming-whisper` | ~3 GB | STT (active) |

GTT counter showed **70.9 GB used** at the moment we checked post-incident
— meaning the GPU memory pool was holding most of the system RAM in
shared mappings, leaving little headroom for the kernel page allocator
when other workloads needed to grow.

### Contributing factor #2: Music streaming added a sustained allocation workload

MA's AirPlay bridge to the Sonos Move runs `ffmpeg` to re-encode audio
into an AirPlay-compatible stream, plus a continuous buffer pipeline. By
14:55 MA had:
- A "Queue Flow stream" for the Sonos
- An AirPlay bridge spinning
- A separate Voice PE stream from earlier (cleared at 14:54:57 but
  containers don't always release memory immediately)

This isn't a huge workload in isolation — a few hundred MB of buffers —
but on top of a system already at 2.4 GB free with 70+ GB locked in GTT,
it was enough to push allocator latencies past the point where the
amdgpu workqueues could keep up.

### Why Bun was the loudest victim

Bun's V8/JSC engine uses W+X JIT pages. When the kernel + amdgpu
allocator start failing under pressure, JIT pages get corrupted while
still mapped as executable. Different threads hit different broken
code paths and segfault at different addresses (first
`0x7F6817209758`, then `0x0`). The Bun panic isn't a Bun bug — it's
**the most easily-detectable symptom** of the host being sick. A
similar process without JIT would likely have just hung silently or
been killed.

### Why this didn't show up as an OOM kill

The kernel's OOM killer fires when a process tries to allocate and the
allocator returns ENOMEM. Under amdgpu GTT pressure, allocations can
**stall** in workqueues for extended periods instead of failing — the
GPU driver's `amdgpu_amdkfd_restore_userptr_worker` (seen hogging CPU
in earlier logs) tries to find pages by evicting GPU mappings rather
than failing the allocation. This means the kernel keeps trying to
satisfy the request, gets backlogged, and eventually some core lockup
or recursion takes the whole system down before OOM kill fires.

## What worked

- **`restart: unless-stopped` on all containers** brought MA, HA, the
  Kronk stack, and all child services back automatically after the
  reboot. Zero manual recovery needed for application state.
- **`systemctl --user enable` on llama units** brought back the LLM
  servers (the ones we wanted) automatically.
- **journald log rotation** (configured 2026-05-23) kept logs around so
  this analysis was possible at all.
- **`sar` daily snapshots** (sysstat default) gave us the memory-pressure
  timeline. Critical for root-causing — without it we'd be guessing.

## What didn't work

- **No real-time memory monitoring.** Operator had no way to know the
  system was getting tight before the crash. `/api/system` returned
  `mem_available: 34 GB` which looked healthy by the naive metric.
- **No alerting on memory pressure.** Even if we'd been watching the
  right number, nothing would have told us to look.
- **The `/resources` dashboard's memory bar only showed `mem_available`**
  — the bucket that hides the problem. `mem_free` was the canary that
  would have shown the cliff coming.
- **No discipline around stopping unused llama services.** We enabled
  `talkie`, `devstral-q4`, and `mistral-nemo` for specific reasons
  (talkie agent, coding tools, v2-coord benchmark respectively) and
  never cleaned up. Each ate 10-15 GB GTT indefinitely.

---

## Mitigations applied (today)

1. **Stopped + disabled three unused llama services:**
   ```bash
   systemctl --user disable --now llama-talkie llama-devstral-q4 llama-mistral-nemo
   ```
   GTT usage: **70.9 GB → 15.6 GB** (-55 GB).
   Free RAM: **~2 GB → ~63 GB**.

2. **Extended `/api/system` endpoint** in `orchestrator/main.py` to expose
   `mem_free`, `gtt_total`, and `gtt_used` alongside the existing fields.

3. **Updated `/resources` dashboard** (`orchestrator/static/resources.html`):
   - Added Free-RAM stat with color thresholds:
     - red `< 2 GB` ("low")
     - amber `< 4 GB` ("tight")
     - yellow `< 8 GB`
   - Added GTT stat showing used/total with color thresholds at >75% and >90%
   - Tooltips explain why these matter (linking back to this incident)

## Lessons

1. **On hardware with shared CPU+GPU memory pools (iGPUs), GPU "memory
   reservation" eats into system memory headroom.** Standard monitoring
   focused on `kbavail` will lie because page cache is reclaimable but
   GTT mappings are not. **The right metric is `kbmemfree`.**

2. **`restart: unless-stopped` + Docker boot-enabled is genuinely good
   disaster recovery for our setup.** Both the kronk stack and the
   kronk-ha + kronk-ma stacks recovered automatically. No data loss
   thanks to named volumes.

3. **"Loaded for an experiment, never unloaded" is a recurring failure
   mode.** Three services accumulated 40+ GB of GTT for tasks we no
   longer needed. A periodic hygiene pass would have caught this.

4. **The most loudly-crashing process is rarely the cause.** Bun was
   the obvious failure visible to the operator, but it was downstream
   of a system-wide pressure event. Always check host-level metrics
   before blaming the application.
